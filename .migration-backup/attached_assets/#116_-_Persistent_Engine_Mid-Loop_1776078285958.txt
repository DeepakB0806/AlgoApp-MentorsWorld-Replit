#116 - Persistent Engine Mid-Loop Concurrency Guard (V2)
What & Why
A race condition exists between startPersistentEntry and startPersistentSquareOff. When Square Off closes an open leg, it briefly frees Kotak margin. Background retry loops (Entry, Exit, Rollback) that are sleeping in their setTimeout cycles wake up, see the freed margin, and blindly execute their next leg — creating ghost trades.

Three independent loops are vulnerable:

startPersistentEntry — retries failed entry legs
startPersistentExit — retries failed exit legs per blockType
startPersistentRollback — reverses margin-shortfall legs
All three must be killed the instant startPersistentSquareOff is triggered, and all three attempt() functions must verify their active status immediately upon waking up — before any DB call.

Done looks like
Square Off kills Entry, Exit, and Rollback loops on its first tick
Each loop's attempt() returns immediately on wake-up if its flag was cleared
Zero ghost legs placed after any manual Square Off
Only server/te-kotak-neo-v3.ts is touched — no EL, UI, DB schema, or storage.ts changes
Tasks
Task 1 — Kill all conflicting loops from startPersistentSquareOff
File: server/te-kotak-neo-v3.ts Location: startPersistentSquareOff → attempt(), at the very top before any other statement (~line 1432)

Add:

// --- TASK #116: KILL ALL CONFLICTING LOOPS ON SQUARE-OFF ---
persistentEntryActive.delete(planId);
persistentRollbackActive.delete(planId);
for (const k of persistentExitActive.keys()) {
  if (k.startsWith(`${planId}:`)) persistentExitActive.delete(k);
}
// -----------------------------------------------------------

Why:

persistentEntryActive uses plain planId as key — direct delete is correct.
persistentRollbackActive uses plain planId — direct delete is correct.
persistentExitActive uses composite keys ${planId}:${blockType} (confirmed at line 1476). delete(planId) would silently miss every key. The startsWith loop is required.
