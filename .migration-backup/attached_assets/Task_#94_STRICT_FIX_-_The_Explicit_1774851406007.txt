#94 - Explicit Config Mandate — No Silent Fallbacks

What & Why
In algorithmic trading, silent fallbacks on execution parameters are dangerous. If a critical setting like inter-leg delay or retry interval is missing from the database, the engine must refuse to trade rather than guessing. All four execution variables must be explicitly configured by the user in the UI, and the Trade Engine must hard-abort if any of them are absent.

Done looks like
A new "Inter-Leg Execution Delay (ms)" field appears in the Trading Execution section of General Settings, wired to order_execution_delay_ms exactly like the other four fields (query, useEffect, mutation, Save button)
In the Trade Engine, every silent fallback on the four critical settings is replaced by a hard abort:
order_execution_delay_ms: 4 call sites — removes || 200 fallback → ABORT if missing
squareoff_retry_interval_ms: 5 call sites — removes || 1000 / || 2000 fallbacks → ABORT if missing
margin_shortfall_retry_count: 1 call site — removes || 0 fallback → ABORT if missing
limit_order_buffer_points: 3 additional call sites (exit/rollback paths) — removes || "0" fallback → ABORT if missing (entry path already aborts correctly)
The startup seed block in server/index.ts does NOT seed any of these four settings — they must be explicitly set by the user
ABORT messages follow the existing pattern: "ABORT: <setting_key> is missing from app_settings"
Out of scope
Changing the existing seeds for rollback_api_retry_count and scrip_master_sync_time (those remain)
Any changes to bracket order logic (completely frozen)
Validation of the numeric range beyond checking for presence
