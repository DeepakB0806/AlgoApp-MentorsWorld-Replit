#169 - Full-Stack Trailing Stop Loss (TSL) — Schema + Engine + UI
What & Why
The Trailing Stop Loss engine already exists for NRML trades but has critical gaps:

Product type restriction: TSL fields only save for NRML. BO and MIS trades (Bracket Orders) never get TSL data written to the DB.
Wiring mismatch: The frontend already sends trailingSL config in the payload, but the backend reads from blockConfig.initialSl / blockConfig.trailingStep — not from trailingSL. If a user configures TSL in the "Trailing Stoploss" UI section, the backend silently ignores it.
UI restriction: The TSL panel is gated behind hasMISLegs — so it only appears when productMode is MIS, despite NRML needing it most.
Missing schema columns: tsl_type, tsl_lock_profit, tsl_profit_step don't exist in the DB.
Engine logic gaps: The TSL engine does simple continuous trailing. The new fields require step-based trailing and lock-profit logic.
TSL Type — exact values (user-specified)
The Type dropdown has exactly 3 options:

UI Label	DB value	Meaning
None	"none"	TSL disabled even if switch is on
% of Capital	"percentage_of_capital"	activateAt and trailingStep are % of entry price × quantity
Amount	"amount"	activateAt and trailingStep are absolute INR amounts (same as points)
When tslType = "none": TSL switch appears but no trailing is applied — exit is only on the fixed initialSlPrice. When tslType = "percentage_of_capital": engine converts % values to INR points using entryPrice as base before each SL calculation. When tslType = "amount": values are used directly as price-point distances.

Done looks like
TSL panel in "Exit & Risk Settings" shows for ALL product types (NRML, BO, MIS) — not just MIS
TSL panel has 5 fields in a row: Type (None / % of Capital / Amount) | Activate At | Lock Profit At | When Profit Increases By | Increase TSL By
Type dropdown is the first field, defaulting to "None"
When Type = None, the 4 numeric fields are disabled/greyed out
DB has 3 new columns: tsl_type, tsl_lock_profit, tsl_profit_step — migration applied via db:push
Trade placement (any product type) writes all 7 TSL fields to the DB row when TSL is configured
TSL engine monitors BO and MIS trades in addition to NRML
Step-based trailing: SL only moves when profit increases by tslProfitStep, then moves by trailingStep
Lock profit logic: once profit ≥ tslLockProfit, SL is moved to at least breakeven before step trailing begins
Out of scope
Per-leg TSL overrides (TSL is plan-level)
UI display of live TSL state on the dashboard (Task #138)
Any changes to locked HSI/HSM blocks

Step 6: TSL engine — update TslState and logic
In server/tsl-kotak-neo-v3.ts:

A. Update TslState interface:

interface TslState {
  tradeId: string; symbol: string; action: string; planId: string | null;
  trailingStep: number; currentSlPrice: number; highWaterMark: number; dirty: boolean; // existing
  tslType: string;              // NEW: "none" | "percentage_of_capital" | "amount"
  tslLockProfit: number | null; // NEW
  tslProfitStep: number | null; // NEW
  entryPrice: number;           // NEW: initial fill price for % calculations
  lockAchieved: boolean;        // NEW: set true once lock-profit threshold crossed
}

B. Update registerNewTrail to populate new fields from trade row:

tslType: trade.tslType ?? "none",
tslLockProfit: trade.tslLockProfit ?? null,
tslProfitStep: trade.tslProfitStep ?? null,
entryPrice: trade.highWaterMark ?? trade.ltp ?? trade.price ?? 0,
lockAchieved: false,

C. Update processTick — for each matching trail state:

Resolve effectiveStep (convert % of capital to INR points):

if tslType === "percentage_of_capital":
  effectiveStep = state.entryPrice * (state.trailingStep / 100)
else: (amount or none)
  effectiveStep = state.trailingStep

Lock-profit check (run once, before trailing):

if !state.lockAchieved && state.tslLockProfit !== null && state.tslLockProfit > 0:
  currentProfit = (BUY ? ltp - entryPrice : entryPrice - ltp)
  if currentProfit >= state.tslLockProfit:
    state.currentSlPrice = MAX(state.currentSlPrice, state.entryPrice)  // at least breakeven
    state.lockAchieved = true
    state.dirty = true

Step-based trailing (replace the current continuous update):

if state.tslProfitStep !== null && state.tslProfitStep > 0:
  // Only advance HWM when price moved by tslProfitStep from current HWM
  BUY: if ltp >= state.highWaterMark + state.tslProfitStep:
    state.highWaterMark = ltp
    state.currentSlPrice = ltp - effectiveStep
    state.dirty = true
  SELL: if ltp <= state.highWaterMark - state.tslProfitStep:
    state.highWaterMark = ltp
    state.currentSlPrice = ltp + effectiveStep
    state.dirty = true
else:
  // Continuous trailing (existing behaviour — no tslProfitStep set)
  BUY: if ltp > highWaterMark: update HWM and SL as before
  SELL: if ltp < highWaterMark: update HWM and SL as before

Exit trigger: unchanged — ltp <= currentSlPrice (BUY) or ltp >= currentSlPrice (SELL)

Guard for tslType="none": if state.tslType === "none", skip all trailing logic — only monitor for exit at the fixed initialSlPrice.
