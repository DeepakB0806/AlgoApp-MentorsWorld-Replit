#137 - HSI Fix: src key + relay path fallback
What & Why
Two confirmed bugs preventing HSI from connecting to E41.

Bug 1 — Wrong auth key: source vs src (PRIMARY)
hslib.js (Kotak's official SDK, now confirmed from uploaded file) intercepts the connection object and rewrites the key before transmitting to E41:

// Keys.SOURCE = "source" (input), but output key is "src":
req = {
  type: "cn",
  Authorization: reqJson[Keys.AUTORIZATION],
  Sid: reqJson[Keys.SID],
  src: reqJson[Keys.SOURCE]   // "source" → "src" over the wire
}

E41's /realtime endpoint requires src: "WEB". Our buildAuthMessage sends source: "WEB". E41 sees an unrecognized field → "session message format incorrect".

This is the definitive root cause of the format error. Proven by:

hslib.js source code (uploaded, line 1, HSIWebSocket constructor)
User connected via demo.html with same tokens → "connected" (hslib.js sent src)
Our code with same tokens → "session message format incorrect" (source rejected)
Bug 2 — Relay path stripping (SECONDARY)
The Bangalore relay (ws://168.144.0.191:8080) receives x-target-url: wss://e41.kotaksecurities.com/realtime but may connect to E41 root (/) instead of /realtime. HSM works because its URL has no path to strip.

Even after fixing Bug 1, if the relay drops /realtime, the connection goes to the wrong E41 endpoint. Detecting "session message format incorrect" while usingRelay=true and setting relayFailed=true forces the next reconnect cycle to go direct — exactly as demo.html connects (directly to wss://e41.kotaksecurities.com/realtime).
