What & Why
Two compounding bugs are making TSL dangerous in production and unscalable:

Bug 1 — activateAt misused as initial SL price (confirmed production loss): In te-kotak-neo-v3.ts line 996, trailingSLCfg.activateAt (= 1000, the profit threshold before trailing begins) was used as the fallback initialSlPrice. An option bought at 143.50 got sl=1000 registered. On the first tick (ltp=149.5), the TSL check ltp <= sl → 149.5 <= 1000 → true → immediate forced sell. Confirmed in production: order 260504000332058 (BUY 143.50) immediately closed by TSL at 150.25.

Bug 2 — TSL is blind between 5-second MTM polls, and does not scale: TSL currently gets prices exclusively from MTM monitor Part A — a REST API call per open trade every 5 seconds. For 1000+ users this means thousands of broker REST calls every 5 seconds. HSM already receives live WebSocket LTP ticks for every subscribed symbol at zero extra cost — but none of those ticks are wired to processTick. TSL is blind between polls.

Bug 3 — processTick is O(n_trails) — does not scale: Every processTick(symbol, ltp) call iterates ALL open trails linearly (for...of trails) and skips non-matching symbols. With 10,000 open trails across users, each symbol tick does 10,000 iterations. Must be O(1) via symbol index.
