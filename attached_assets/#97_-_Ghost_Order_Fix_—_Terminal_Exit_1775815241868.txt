Task #97: Relational Error Command Centre (Multi-Tenant Scaling)

What & Why
As the platform scales to 50+ users, the variety of Kotak API and RMS errors will multiply. A hardcoded array or a flat comma-separated string for error handling is unscalable. We are building a true Relational Error Routing Engine. This involves a dedicated SQL table where administrators can log exact Kotak error patterns, assign them specific system actions (like terminal_close), and toggle them globally for all users.

Done looks like

Accurate UI Data: The Rejection Reason and Time columns in the Order Panel show Kotak's actual rejRsn and ordDtTm fields.

Schema & DB: A new error_routing table exists in the PostgreSQL database.

Settings UI: The Admin Settings page features a data table to Create, Read, Update, and Delete (CRUD) error routing rules.

Dynamic Trade Engine: closeTrade() queries the active error routes from the DB. If a broker error matches a pattern, it executes the assigned action (e.g., terminating the zombie loop).

Manual Override: A "Force Close" button remains available on close_failed trades for unmapped errors.

Task 4 — Trade Engine Integration
File: server/te-kotak-neo-v3.ts (inside closeTrade failure handler)
Replace the hardcoded checks with a database query:

TypeScript
const activeRoutes = await storage.getActiveErrorRoutes();
const errStr = String(orderResult.error || "").toLowerCase();

const matchedRoute = activeRoutes.find(route => 
  errStr.includes(route.errorPattern.toLowerCase())
);

if (matchedRoute) {
  if (matchedRoute.actionType === "terminal_close") {
    const terminalMsg = `[TE] Terminal rejection mapped by rule #${matchedRoute.id}: "${orderResult.error}". Marking as closed.`;
    console.warn(terminalMsg);
    await logPFL(plan.id, trade.entryLegId || "N/A", "force_closed", terminalMsg);
    
    const closedUpdate = await storage.updateStrategyTrade(trade.id, {
      status: "closed", pnl: 0, exitedAt: new Date(), updatedAt: new Date(), 
      exitPrice: trade.ltp || trade.price || 0, exitAction: plan.exitAction || "SELL"
    });
    
    tradingCache.invalidateOpenTrades(trade.planId);
    return closedUpdate || trade;
  }
  // Future action types can be handled here
}

Why this is a Game Changer
With this implementation, you are no longer just fixing a bug. You are building an internal administrative tool. When a new user joins and encounters a weird margin error specific to their Kotak account tier, you will see it in the logs, go to your Settings UI, add the exact error string, map it to "terminal_close", and instantly resolve it for them and everyone else on the platform forever.
