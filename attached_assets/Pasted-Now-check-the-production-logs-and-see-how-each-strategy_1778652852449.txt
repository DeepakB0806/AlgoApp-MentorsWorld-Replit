Root Cause #2 — MTM = -1,532,326 with threshold -500
The user is correct — HSM price is the proximate cause, but the deeper issue is the fill price stored at entry.

Trace through computePlanMTM:

const legMTM = trade.action === "BUY"
  ? (ltp - entryPrice) * qty    // ← ltp from HSM/getPrice
  : (entryPrice - ltp) * qty;

For MTM = -1,532,326 with qty = 65:

(ltp − entryPrice) × 65 = −1,532,326
→ (ltp − entryPrice) = −23,574

The only way this number appears: entryPrice stored in the DB is the NIFTY underlying spot price (~23,624.5) rather than the option premium (~50–240 rupees).

This happens through getFillPrice's fallback chain in te-kotak-neo-v3.ts. When EL.getOrderHistory is empty or fails (we see getOrderHistory empty for ... retrying in 1s... in the logs repeatedly), the fill price falls back to currentPrice — which is the NIFTY index spot price from the webhook signal payload (price=23624.5). That spot price gets stored in strategy_trades.price as the "entry price" of the option leg.

So for NIFTY2651923000PE (May 19 PE), the stored entry price is ₹23,624 instead of, say, ₹50. When the MTM monitor fetches the actual option LTP (say ₹50 from HSM or REST quote):

MTM = (50 − 23,624) × 65 = −23,574 × 65 = −1,532,310  ≈ −1,532,326

That's the exact number in the log. The HSM price is correct for the option — it's the stored entry price that is wrong (spot price used as fill price fallback).
